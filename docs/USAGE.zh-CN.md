# KarMind 使用指南

[English guide](USAGE.md)

KarMind 用来把零散素材逐步编译成一个可维护的 wiki。核心飞轮是：

```text
收集 -> 编译 -> 提问 -> 回填 -> 健康检查 -> 继续收集
```

## 1. 设置 KarMind

1. 打开 **设置 -> KarMind**。
2. 设置 **LLM API 地址**，需要兼容 OpenAI API。
3. 选择或创建 **API key** secret。
4. 确认这些文件夹：
   - `raw/`：保存你收集的原始素材。
   - `wiki/`：保存 KarMind 生成和维护的知识页面。
   - `skills/`：可选的声明式 skills，每个 skill 一个文件夹。
5. 建议把 **默认权限** 保持为 **基础问答**，除非你希望新对话默认就能读写笔记。

KarMind 只会在你运行 LLM 相关操作时，把必要内容发送给你配置的 LLM API。

## 2. 收集 raw 素材

把原始素材放进 `raw/`。适合放入 raw 的内容包括：

- 网页剪藏。
- 论文笔记。
- 阅读笔记。
- 会议记录。
- 代码或项目资料。
- 个人研究碎片。

raw 笔记应当被看作“源材料”。KarMind 会读取它们，但长期维护的知识应主要放在 `wiki/` 中。

对于网页文章，推荐使用 [Obsidian Web Clipper](https://obsidian.md/clipper) 浏览器插件。你可以把网页剪藏成 Markdown，保存到 `raw/` 下，然后在准备整合进知识库时运行 `/compile`。

## 3. 编译 raw 笔记

运行：

```text
/compile
```

KarMind 会读取 `raw/` 中发生变化的文件，发送给你配置的 LLM，并把结构化页面写入 `wiki/`。

编译可能消耗大量 token，尤其是 raw 文件很长或一次变化了很多文件时。如果你正在测试新的模型服务或提示词配置，建议先用少量素材试跑。

只有当你想强制重新处理所有 raw 文件时，才使用：

```text
/compile --force
```

普通 `/compile` 会通过 `wiki/.karmind/source-manifest.json` 跳过未变化文件，从而节省 token。

适合使用编译的情况：

- 你添加了新素材。
- 你修改了已有 raw 笔记。
- 你希望把原始材料转成主题页面、摘要和交叉链接。

不要用 `/compile` 来做临时问题回答。临时提问应该用 `/qa`。

## 4. 基于 wiki 提问

运行：

```text
/qa 你的问题
```

KarMind 会扫描相关 `wiki/` 页面，把选中的上下文发送给 LLM，并在对话中回答。

适合使用 QA 的情况：

- 你想知道当前 wiki 已经掌握了什么。
- 你想跨多个 wiki 页面做综合分析。
- 你正在探索一个想法，但还不确定是否值得保存。

如果回答有长期价值，再用 `/backfill` 保存。

## 5. 回填有价值的结果

运行：

```text
/backfill 要保存的内容
```

回填会让 LLM 判断如何把内容合并进 wiki。它可能创建新页面，也可能更新已有页面，然后更新 `_index.md` 并追加 `log.md`。

适合使用回填的情况：

- 某次 `/qa` 的回答值得长期保留。
- 你生成了有用的对比、总结、计划或洞见。
- 健康检查发现缺失概念后，你想补充对应页面。
- 你希望把一个精炼后的结论合并进已有 wiki 页面。

示例：

```text
/backfill 保存这个结论：三数之和通常通过排序、固定一个数、再使用双指针解决。请关联到 [[双指针]]、[[去重策略]] 和 [[LeetCode]]。
```

不适合使用回填的情况：

- 内容还是未经整理的原始材料。这种情况应该先放进 `raw/`，再运行 `/compile`。
- 你不希望 KarMind 修改 wiki 页面。
- 你需要严格的 diff 审核。KarMind 当前会记录文件操作，但还没有完整的“应用前 diff 审批”步骤。

## 6. 运行健康检查

运行：

```text
/health
```

KarMind 会检查断链、孤立页面、缺失概念、图谱结构问题，以及 LLM 发现的一致性缺口。

报告会保存到：

```text
wiki/_reports/health/
```

适合使用健康检查的情况：

- wiki 变大后开始混乱。
- 你看到很多断链。
- 你想知道缺哪些概念页面。
- 你想决定下一步应该回填或编译什么。

## 7. 使用 skills

运行：

```text
/skills
```

可以查看可用 skills。

运行：

```text
/skill <id> [args...]
```

可以手动执行某个 skill。

声明式 vault skills 放在：

```text
skills/<skill-name>/skill.md
```

KarMind 也会把已启用的 skills 暴露给模型，让模型可以在对话中主动建议使用相关 skill。

## 8. 权限和安全

KarMind 有两种会话权限：

- **基础问答**：更安全的默认权限，适合普通聊天和 `/qa`。
- **增强笔记**：允许读写笔记，包含 `/compile`、`/backfill`、`/health` 和 `/skill`。

敏感操作会弹出确认提示。编译可能消耗较多 token。增强笔记操作可能创建或更新文件。

## 9. 推荐工作流

一次正常的知识整理流程可以是：

1. 把素材加入 `raw/`。
2. 运行 `/compile`。
3. 在 Obsidian 中浏览生成的 `wiki/` 页面。
4. 用 `/qa` 问具体问题。
5. 用 `/backfill` 保存有价值回答。
6. 当 wiki 变大后运行 `/health`。
7. 优先修复或回填最高价值的问题。

目标不是把每一次聊天都保存下来，而是只回填真正能增强长期 wiki 的结果。

## 10. 常见问题

- 如果 `/compile` 没有动静，可能是 raw 文件没有变化。确实要全部重编译时使用 `/compile --force`。
- 如果命令需要更高权限，请确认增强权限提示。
- 如果 LLM 请求失败，请检查 API 地址、模型名、API key secret，以及服务商的网络限制。
- 如果 wiki 链接有高亮但不能打开，请更新到最新构建后重载插件。
- 如果生成的 markdown 格式不理想，可以手动编辑 wiki 页面，或让 KarMind 回填修正版。
